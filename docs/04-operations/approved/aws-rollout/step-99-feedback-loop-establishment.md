# 99단계. 운영/개발 피드백 루프 구축

## 목적
- 운영팀과 개발팀 간의 정기적인 피드백 채널을 마련해 개선 사항을 신속하게 반영한다.
- 91단계 온보딩, 95단계 릴리즈 기록, 97단계 정기 회의에서 나온 인사이트를 공유한다.

## 선행조건
- [97단계 정기 리뷰 회의](docs/04-operations/approved/aws-rollout/step-97-regular-review-meetings.md)의 일정과 아젠다가 확정되어 있어야 한다.
- 협업 도구(Slack, Teams, Wiki 등)와 이슈 추적 시스템(Jira 등)이 운영 중이어야 한다.
- 온보딩 설문(91단계), 보안 교육 피드백(92단계), 자동화 개선 요청(93단계)이 수집되고 있어야 한다.

## 필요한 입력 자료
- 최근 릴리즈 이슈 목록, 포스트모템 후속 조치, 사용자 피드백 요약.
- 피드백 채널 운영 정책 초안과 커뮤니케이션 가이드라인.
- 참여자 명단과 역할(운영, 개발, 제품, 보안 대표).

## 상세 절차
1. **피드백 채널 설계**
   - 정기 미팅, 비동기 게시판, 설문 등 채널 유형을 정의한다.
   - 익명 제안 수단 여부와 응답 SLA를 결정한다.
2. **프로세스 및 책임 할당**
   - 피드백 수집→분류→우선순위 지정→조치→결과 공유까지 흐름을 정의한다.
   - 채널 운영자, triage 담당자, 실행 책임자를 지정한다.
3. **도구 구성 및 템플릿 마련**
   - 협업 도구에 피드백 폼/템플릿을 생성하고, 카테고리(기능, 인프라, 보안 등)를 설정한다.
   - 94단계 변경 관리와 연계해 피드백이 변경 요청으로 전환될 조건을 명확히 한다.
4. **커뮤니케이션 및 참여 유도**
   - 팀 전체에 피드백 루프 목적과 사용 방법을 안내한다.
   - 91·92단계 교육에서 참여 방법을 소개하고 참여 인센티브를 설계한다.
5. **성과 측정 및 개선**
   - 처리 시간, 해결율, 만족도 등을 지표화해 90단계 KPI에 추가한다.
   - 정기적으로 채널 효과성을 평가하고 프로세스를 개선한다.

## 의사결정 포인트
- 피드백 응답 SLA를 몇 일로 설정할지.
- 익명 피드백을 허용할지 여부와 악용 방지 대책.
- 피드백 처리 결과를 어떤 채널로 공유할지(정기 회의 vs. 대시보드).

## 체크리스트
- [ ] 피드백 채널과 프로세스가 정의되고 책임자가 지정되었다.
- [ ] 도구 설정과 템플릿이 준비되었다.
- [ ] 팀에 커뮤니케이션이 완료되고 참여 방법이 안내되었다.
- [ ] 처리 지표가 정의되고 정기적으로 리뷰된다.

## 산출물 및 보관 위치

- 피드백 수집 대시보드(`dashboards/feedback/status.json`).
- 처리 현황 보고(`reports/feedback/feedback-summary-YYYYMM.xlsx`).
