# 44단계. 코드 리포지토리와 CI 연동

## 목적
- 35단계에서 정비한 코드 저장소와 43단계에서 구성한 CI 인프라를 연결하여 자동 빌드 트리거를 활성화한다.
- 브랜치/태그 전략, 승인 정책을 기반으로 파이프라인 실행 조건과 알림을 설정한다.

## 선행조건
- [35단계 코드 저장소 정리](docs/04-operations/approved/aws-rollout/step-35-repository-governance.md).
- [43단계 CI 도구 인프라 구성](docs/04-operations/approved/aws-rollout/step-43-ci-infrastructure-setup.md).
- 3단계 버전 전략, 27단계 파이프라인 요구사항 문서.

## 필요한 입력 자료
- 저장소 URL, 브랜치 보호 규칙, 필수 리뷰어 목록.
- 파이프라인 정의 파일 템플릿(예: YAML 워크플로우).
- 알림 채널(Slack, 이메일) 및 접근 토큰/웹훅 정보.

## 상세 절차
1. **연동 방식 선택**
   - SaaS 기반 CI의 경우 리포지토리 앱 설치 또는 OIDC 연동 설정을 완료한다.
   - 셀프 호스트 CI는 Git Webhook, SCM 플러그인, 폴링 방식 중 하나를 선택한다.
2. **트리거 규칙 구성**
   - 3단계에서 정의한 브랜치 전략에 따라 main/release/feature 브랜치의 트리거 조건을 설정한다.
   - 태그 릴리스, Pull Request, 스케줄 기반 실행을 구분하여 설정한다.
3. **파이프라인 정의 파일 배치**
   - 저장소 루트 또는 지정 디렉터리에 파이프라인 YAML을 배치하고 버전 관리한다.
   - 공통 템플릿(27단계 요구사항)을 재사용하여 일관성을 유지한다.
4. **권한 및 시크릿 설정 확인**
   - 41단계 IAM 역할 AssumeRole 권한이 CI 런너에서 사용 가능한지 검증한다.
   - 리포지토리 시크릿/변수에 22, 38단계에서 정한 보안 정책을 적용한다.
5. **알림 및 감사 구성**
   - 빌드 실패/성공 알림을 26단계 모니터링 또는 Incident 채널과 연동한다.
   - 감사 목적으로 파이프라인 실행 로그를 중앙 저장소(S3/CloudWatch Logs)에 보존한다.
6. **연동 테스트**
   - 샘플 커밋 또는 Pull Request를 생성하여 파이프라인이 자동으로 실행되는지 확인한다.
   - 실패 시 로그를 분석하고 트리거/권한 설정을 수정한다.

## 의사결정 포인트
- 브랜치별 파이프라인 분리 여부 및 실행 빈도.
- 파이프라인 정의 파일을 모노레포 vs. 서비스별 분리로 관리할지 여부.
- 알림 채널과 Incident 관리 도구 통합 범위.

## 체크리스트
- [ ] 리포지토리와 CI 도구 간 인증이 성공적으로 구성되었다.
- [ ] 브랜치/태그/PR 트리거 규칙이 문서화되었다.
- [ ] 파이프라인 정의 파일이 저장소에 커밋되어 검토를 거쳤다.
- [ ] 알림 및 로그 보존 설정이 완료되었다.

## 산출물 및 보관 위치
- 파이프라인 정의 파일 (`.github/workflows/*.yml`, `ci/pipelines/*.yaml`).
- 연동 구성 문서 및 다이어그램 (`docs/04-operations/approved/aws-rollout/ci-integration.md`).
- 테스트 실행 로그 및 결과 보고서 (`docs/reports/ci-integration-test.md`).
