# 22단계. 런타임 환경 변수/시크릿 관리 방안 결정

## 목적
- 애플리케이션 실행에 필요한 환경 변수와 시크릿(키, 토큰, 인증 정보)을 안전하게 저장·배포하는 전략을 확정한다.
- 배포 파이프라인과 운영 절차에서 시크릿을 노출하지 않도록 통제한다.

## 선행조건
- 7단계 보안 및 규제 요구사항 분석.
- 8단계 거버넌스 원칙과 태그 정책.
- 20단계 컴퓨팅 서비스, 21단계 아티팩트 전략 결정.

## 필요한 입력 자료
- 환경별(개발/스테이징/운영) 시크릿 목록과 민감도 분류.
- 규제 준수 요구사항(암호화, 접근 로깅, 회전 주기).
- 운영팀과 개발팀의 접근 권한 요구사항.

## 상세 절차
1. **시크릿 자산 식별 및 분류**
   - 데이터베이스 자격 증명, API 키, OAuth 비밀, 타사 통합 토큰을 목록화한다.
   - 민감도 및 회전 주기를 분류하고, 보존 기간을 설정한다.
2. **관리 도구 평가**
   - AWS Secrets Manager, AWS Systems Manager Parameter Store, HashiCorp Vault, 외부 HSM 등 후보를 비교한다.
   - 비용(6단계), 운영 복잡도, 감사 로깅, 자동 회전 기능을 기준으로 선택한다.
3. **키 관리 및 암호화 설계**
   - KMS 키 정책, 키 별 태그, 자동 회전 여부를 정의한다.
   - 멀티 계정 시 Cross-account KMS 사용 전략을 설계한다.
4. **프로비저닝 및 접근 정책 수립**
   - IAM 정책, VPC 엔드포인트, 리소스 기반 정책을 설정한다.
   - 최소 권한 원칙(12단계)과 연계하여 읽기/쓰기/관리 권한을 구분한다.
5. **파이프라인 통합 설계**
   - 27~30단계 CI/CD 파이프라인에서 시크릿을 주입하는 절차를 정의한다.
   - 시크릿이 로그나 아티팩트에 기록되지 않도록 마스킹 규칙을 설정한다.
6. **운영 절차 및 감사 계획**
   - 회전 스케줄, 비상 접근 절차, 감사 로그 검토 주기를 문서화한다.
   - 인시던트 대응 시 시크릿 무효화 및 재발급 프로세스를 정의한다.

## 의사결정 포인트
- 관리형 시크릿 저장소 vs. 자체 Vault 운영.
- 계정/리전에 따라 시크릿을 분리할지 통합할지 여부.

## 체크리스트
- [ ] 환경별 시크릿 인벤토리가 작성되었다.
- [ ] 시크릿 관리 도구 및 KMS 전략이 선택되었다.
- [ ] CI/CD 및 운영 절차 통합 방안이 문서화되었다.
- [ ] 회전·감사 정책이 승인되었다.

## 산출물 및 보관 위치
- 시크릿 관리 아키텍처 문서 (`docs/security/runtime-secrets-plan.md`).
- KMS 및 IAM 정책 템플릿 (`infra/security/kms/`).
- 회전 및 비상 대응 Runbook (`docs/runbooks/secret-rotation.md`).
