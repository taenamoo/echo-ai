# 35단계. 코드 저장소 정리

## 목적
- 배포 대상 코드베이스 구조를 정비하여 협업 효율과 파이프라인 일관성을 확보한다.
- 브랜치/태그 정책과 보호 규칙을 설정해 품질과 감사 요구를 충족한다.

## 선행조건
- 3단계 버전 전략, 4단계 IaC 도입 결정.
- 27단계 CI/CD 요구사항, 31단계 품질 게이트 기준.
- 34단계 승인 절차 결과.

## 필요한 입력 자료
- 현재 저장소 구조, 서브모듈/패키지 의존성 지도.
- 릴리스 주기, 팀별 개발 흐름, 코드 리뷰 정책.
- 보안 요구사항(7단계) 및 감사 로그 보관 요구.

## 상세 절차
1. **저장소 구조 평가**
   - 모노레포 vs. 폴리레포 현황을 분석하고 장단점을 검토한다.
   - IaC, 애플리케이션, 운영 스크립트 등의 디렉터리 구조를 표준화한다.
2. **브랜치 전략 적용**
   - 3단계에서 정의한 브랜치 모델(예: trunk-based, GitFlow)을 실제 저장소에 반영한다.
   - 필수 브랜치에 보호 규칙(리뷰 수, CI 통과)을 설정한다.
3. **태그/릴리스 관리 표준화**
   - 릴리스 태그 네이밍 규칙, 체인지로그 작성 프로세스를 문서화한다.
   - 30단계 빌드 파이프라인에서 태그 기반 배포 자동화를 구성한다.
4. **보안 및 접근 제어 강화**
   - 12단계 IAM 설계와 연동해 저장소 접근 권한을 점검한다.
   - 시크릿 스캔, 서드파티 앱 접근 권한을 검토한다.
5. **자동화 및 템플릿 정비**
   - 이슈/PR 템플릿, 커밋 메시지 규칙, 코드 소유자 파일 등을 업데이트한다.
   - 27단계 파이프라인 구성을 저장소 내 표준 위치에 정리한다.
6. **문서화 및 교육**
   - 저장소 구조도, 브랜치/태그 정책을 문서화하여 공유한다.
   - 신규 기여자 온보딩 자료를 업데이트한다.

## 의사결정 포인트
- 모노레포 통합 여부 및 팀 간 협업 영향.
- 필수 리뷰어/승인자 지정 기준과 자동화 수준.

## 체크리스트
- [ ] 저장소 구조 및 표준 디렉터리가 정리되었다.
- [ ] 브랜치 보호 및 태그 정책이 설정되었다.
- [ ] 접근 권한과 보안 스캔 구성이 검토되었다.
- [ ] 문서화 및 온보딩 자료가 최신화되었다.
