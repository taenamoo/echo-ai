# 37단계. 리소스 태깅 규칙 상세화

## 목적
- AWS 리소스에 적용할 표준 태그 체계를 정의하여 비용 관리, 보안, 운영 자동화에 활용한다.
- 태깅 정책을 IaC 및 파이프라인에 통합해 일관성을 확보한다.

## 선행조건
- 8단계 거버넌스 원칙 수립 결과.
- 21단계 아티팩트 전략, 26단계 모니터링 체계, 36단계 비용 모니터링 초안.
- 34단계 승인 절차와 35단계 저장소 정리.

## 필요한 입력 자료
- 조직 구조(팀/프로젝트/제품)와 비용 배분 요구.
- 규제 또는 보안 감사에서 요구하는 태그 항목 목록.
- IaC 템플릿, 배포 스크립트, 수동 프로비저닝 절차.

## 상세 절차
1. **태그 카탈로그 정의**
   - 필수 태그(예: Project, Environment, Owner, CostCenter, DataClass)를 지정한다.
   - 선택/옵션 태그와 사용 금지 태그 리스트를 작성한다.
2. **명명 규칙 및 검증 로직 설계**
   - 키/값 포맷(대소문자, 구분자), 길이 제한, 허용 문자 규칙을 정한다.
   - 규칙 위반 시 경고/차단 정책을 31단계 품질 기준과 연계한다.
3. **적용 범위 및 책임 분담 명시**
   - 자동화(IaC, 파이프라인) vs. 수동 콘솔 작업 시 책임 팀을 정의한다.
   - 비관리형 리소스(서드파티, SaaS) 태깅 대안을 논의한다.
4. **자동화 및 감사 구성**
   - AWS Config, Service Catalog, Tag Policies, SCP 등을 활용해 정책을 적용한다.
   - 태그 누락 감지 Lambda/Step Functions 워크플로를 설계한다.
5. **교육 및 변경 관리**
   - 태깅 정책 가이드와 예시를 문서화하고 공유한다.
   - 변경 시 승인 절차(34단계)와 커뮤니케이션 계획을 정의한다.

## 의사결정 포인트
- 필수 태그 세트와 예외 허용 범위.
- 태그 정책 적용 도구(Tag Policies vs. Config Rules) 선택.

## 체크리스트
- [ ] 태그 카탈로그와 규칙이 문서화되었다.
- [ ] 자동화/감사 도구 구성이 계획되었다.
- [ ] 책임 분담과 교육 계획이 수립되었다.
- [ ] 태그 정책 변경 관리 절차가 정의되었다.
