# 4단계. IaC(Infra as Code) 도입 여부 결정

## 목적
- 인프라를 코드로 관리할지 여부와 사용할 도구(Terraform, AWS CDK, CloudFormation 등)를 결정한다.
- 선택된 도구에 맞춰 리포지토리 구조와 협업 절차를 정의한다.

## 선행조건
- 2단계 아키텍처 정리 자료.
- 3단계에서 확정된 버전 관리 전략 및 리포지토리 운영 정책.

## 필요한 입력 자료
- 현재 인프라 프로비저닝 방식과 운영팀 역량.
- 도입 가능한 IaC 도구 목록과 라이선스/비용 정보.
- 보안 및 컴플라이언스 요구사항(코드 리뷰, 감사 추적 등).

## 상세 절차
1. **요구사항 분석**
   - 자동화 수준, 멀티 계정 관리 필요성, 모듈화 요구사항을 정의한다.
   - 보안/감사 요구사항이 IaC 도구 선택에 영향을 주는지 확인한다.
2. **도구 비교 평가**
   - Terraform, AWS CDK, CloudFormation, Pulumi 등 후보 도구를 기능, 팀 역량, 커뮤니티 지원 측면에서 비교한다.
   - 멀티 클라우드, 정책 코드화 기능이 필요한지 여부를 평가한다.
3. **파일/모듈 구조 설계**
   - 모듈 배치 전략(예: 환경별 디렉터리, 서비스별 모듈)을 설계한다.
   - 3단계에서 정의한 브랜치 전략을 적용하여 PR 기반 변경 흐름을 설계한다.
4. **상태 관리 및 백엔드 결정**
   - Terraform 사용 시 S3 + DynamoDB Lock 등 상태 저장소를 정한다.
   - AWS CDK라면 스택 구성과 배포 명령어 표준을 정리한다.
5. **보안/품질 가드레일 정의**
   - 코드 리뷰, Static Analysis(TFLint, cdk-nag 등), 정책 적용 방식을 정한다.
   - 비밀정보는 외부 시스템(Secrets Manager 등)과 연계하도록 규칙을 만든다.
6. **PoC 수행 및 결정 확정**
   - 소규모 범위로 PoC를 수행하여 팀 역량과 도구 적합성을 검증한다.
   - PoC 결과를 바탕으로 최종 도입 여부와 운영 계획을 승인받는다.
7. **운영 가이드 문서화**
   - 디렉터리 구조, 명령어, 배포 프로세스를 문서화하여 저장한다.
   - 온보딩 가이드와 CI/CD 파이프라인 요구사항에 반영한다.

## 의사결정 포인트
- 멀티계정/멀티리전에 대한 관리가 필요한지 여부.
- Terraform Enterprise, Spacelift 등 관리형 플랫폼 도입 필요성.

## 체크리스트
- [ ] IaC 도구 후보에 대한 비교 평가 문서가 작성되었다.
- [ ] 상태 관리 및 보안 가드레일 전략이 정의되었다.
- [ ] PoC 결과와 최종 도입 여부가 승인되었다.
- [ ] 리포지토리 구조 및 운영 가이드가 문서화되었다.

## 산출물 및 보관 위치

- 리포지토리 구조 다이어그램과 운영 가이드 (예: `infra/README.md`).
