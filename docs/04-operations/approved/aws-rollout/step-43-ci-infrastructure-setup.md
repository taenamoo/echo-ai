# 43단계. CI 도구 인프라 구성

## 목적
- 28단계에서 선정한 CI 도구(GitHub Actions, CodeBuild, Jenkins 등)를 설치하거나 워크로드 계정에 구성하여 빌드 자동화를 구현한다.
- CI가 AWS 리소스에 접근할 수 있도록 네트워크, 자격 증명, 보안 구성을 확립한다.

## 선행조건
- [28단계 CI 도구 선택](docs/04-operations/approved/aws-rollout/step-28-ci-tool-selection.md).
- [41단계 IAM 역할/사용자 생성 및 검증](docs/04-operations/approved/aws-rollout/step-41-iam-provisioning.md).
- 26단계 모니터링 프레임워크, 31단계 품질 기준.

## 필요한 입력 자료
- CI 도구 설치 가이드 또는 SaaS 설정 매뉴얼.
- 네트워크/VPC 설계 문서(14~19단계), 방화벽/보안 그룹 규칙.
- CI가 접근해야 하는 리포지토리, 아티팩트 저장소, 시크릿 목록.

## 상세 절차
1. **배포 모델 결정**
   - SaaS 사용 시 IP 허용 목록, OIDC 구성 등 접근 제어를 검토한다.
   - 셀프 호스팅 시 VPC, 서브넷, 보안 그룹, OS 이미지를 확정한다.
2. **인프라 프로비저닝**
   - IaC 또는 설치 스크립트를 사용해 CI 서버/서비스를 생성한다.
   - 고가용성이 필요한 경우 멀티 AZ 또는 오토 스케일링 구성을 고려한다.
3. **자격 증명 통합**
   - 41단계에서 만든 IAM 역할을 OIDC/OAuth 또는 액세스 키 없이 AssumeRole 하도록 설정한다.
   - 시크릿 관리 도구(22/38단계)와 연동하여 자격 증명 노출을 방지한다.
4. **네트워크 및 보안 설정**
   - 소스 리포지토리, 아티팩트 저장소, 패키지 레지스트리로의 아웃바운드 트래픽을 허용한다.
   - CI 서버에 대한 접근 제어(SSH, 콘솔 접근)를 제한하고 감사 로그를 활성화한다.
5. **도구 플러그인 및 런너 구성**
   - 필요한 빌드 에이전트/런너 이미지를 준비하고 표준 빌드 환경을 정의한다.
   - 캐시, 병렬 실행, 자원 할당 정책을 문서화한다.
6. **검증 및 모니터링 연계**
   - 샘플 파이프라인을 실행하여 성공적으로 완료되는지 확인한다.
   - 26단계 모니터링 도구와 통합하여 빌드 실패 알람을 설정한다.

## 의사결정 포인트
- SaaS vs. 셀프 호스팅 선택과 보안 요구사항 충족 여부.
- CI 작업자(런너)를 온디맨드로 생성할지 상시 실행할지 여부.
- 네트워크 격리 수준(VPC, 프록시, Bastion) 결정.

## 체크리스트
- [ ] CI 인프라가 IaC 또는 문서화된 절차로 재생성 가능하다.
- [ ] IAM 역할 및 시크릿 통합이 완료되어 키가 노출되지 않는다.
- [ ] 네트워크/보안 설정이 검증되었고 감사 로그가 활성화되었다.
- [ ] 샘플 파이프라인 실행 결과가 성공적으로 기록되었다.

## 산출물 및 보관 위치
- CI 인프라 IaC 템플릿 (`infra/ci/`), 설치 스크립트 (`scripts/ci-bootstrap/`).
- CI 아키텍처 및 운영 가이드 (`docs/runbooks/ci-operations.md`).
- 샘플 파이프라인 정의 파일 (`.github/workflows/`, `ci/pipelines/`).
