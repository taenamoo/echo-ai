# 30단계. 아티팩트 빌드 파이프라인 설계

## 목적
- 소스 코드 변경을 검증하고 배포 가능한 아티팩트를 생성하는 빌드 파이프라인을 설계한다.
- 보안 스캔, 테스트, 이미지 서명 등 품질 게이트를 포함하여 반복 가능한 빌드 프로세스를 정의한다.

## 선행조건
- 21단계 아티팩트 전략, 22단계 시크릿 관리 방안.
- 27단계 파이프라인 요구사항, 28단계 CI 도구, 29단계 CD 도구 결정.
- 31단계 보안 스캔 기준(초안) 및 32단계 배포 전략 검토.

## 필요한 입력 자료
- 코드 레포지토리, Dockerfile/빌드 스크립트, IaC 템플릿.
- 테스트 스위트, 보안 스캔 도구 구성, 서명/증명 요구사항.
- 환경별 설정 파일, 시크릿 참조 정보.

## 상세 절차
1. **파이프라인 단계 설계**
   - 체크아웃 → 의존성 설치 → 정적 분석 → 단위 테스트 → 빌드 → 보안 스캔 → 이미지/패키지 생성 → 서명 → 업로드 순으로 설계한다.
   - 병렬 실행과 캐시 전략을 정의하여 빌드 시간을 최적화한다.
2. **환경 변수 및 시크릿 주입 설계**
   - 22단계에서 결정한 시크릿 저장소에서 필요한 값만 주입한다.
   - 빌드 로그에 시크릿이 노출되지 않도록 마스킹 정책을 적용한다.
3. **빌드 아티팩트 생성 및 저장소 연동**
   - 21단계에서 정의한 레지스트리/저장소에 이미지를 푸시하거나 패키지를 업로드한다.
   - 태그/버전 규칙, 라이프사이클 정책과 일치하도록 구현한다.
4. **보안 및 품질 게이트 통합**
   - 31단계에서 정의할 SAST/DAST/컨테이너 스캔 도구를 미리 통합한다.
   - 빌드 실패 기준, 경고 임계값, 승인 프로세스를 설정한다.
5. **빌드 결과 검증 및 서명**
   - SBOM 생성, 서명(cosign 등), 무결성 증명 저장 절차를 마련한다.
   - 빌드 메타데이터(커밋 SHA, 빌드 번호)를 기록한다.
6. **로그/지표 수집 및 알림**
   - 26단계 모니터링 도구와 연계하여 빌드 시간, 실패율, 큐 대기시간 등을 수집한다.
   - 실패 시 알림 채널로 자동 전파하도록 설정한다.
7. **템플릿화 및 재사용성 확보**
   - 서비스별 공통 템플릿, 재사용 가능한 워크플로 구성요소를 정의한다.
   - 파이프라인 코드 저장소 구조(예: `.github/workflows/`, `infra/ci/templates/`)를 결정한다.

## 의사결정 포인트
- 단일 모노레포 파이프라인 vs. 서비스별 파이프라인 템플릿.
- 빌드 캐시(원격 캐시, Docker layer cache) 및 인프라 선택.

## 체크리스트
- [ ] 빌드 단계와 품질 게이트가 정의되었다.
- [ ] 시크릿 주입 및 보안 정책이 반영되었다.
- [ ] 아티팩트 저장소 연동과 버전 규칙이 설정되었다.
- [ ] 모니터링·알림 통합이 구성되었다.
- [ ] 파이프라인 템플릿/코드가 준비되었다.

## 산출물 및 보관 위치
- 빌드 파이프라인 설계 문서 (`docs/ci-cd/build-pipeline-design.md`).
- 파이프라인 코드 템플릿 (`infra/ci/templates/`).
- 품질 게이트 설정 파일 (`infra/ci/policies/`).
