# 63단계. CI 파이프라인 전체 실행 리허설

## 목적
- 코드 커밋부터 빌드, 테스트, 아티팩트 업로드까지 CI 흐름을 실제와 동일하게 리허설하여 문제를 사전 발견한다.
- 56~58단계에서 구축한 품질 게이트와 모니터링이 기대대로 동작하는지 확인한다.

## 선행조건
- [27단계 CI 요구사항](docs/04-operations/approved/aws-rollout/step-27-cicd-requirements.md) 및 [43단계 CI 인프라 구축](docs/04-operations/approved/aws-rollout/step-43-ci-infrastructure-setup.md) 완료.
- [46단계 테스트 스위트 통합](docs/04-operations/approved/aws-rollout/step-46-test-suite-integration.md), [47단계 품질 게이트 구성](docs/04-operations/approved/aws-rollout/step-47-quality-gates-configuration.md) 적용.
- [56단계 빌드 검증](docs/04-operations/approved/aws-rollout/step-56-initial-build-validation.md) 결과를 기준으로 삼을 수 있도록 아티팩트 저장소 구성이 완료되어야 한다.

## 필요한 입력 자료
- 대표 브랜치에 병합 예정인 코드 변경 사항 또는 더미 커밋.
- CI 파이프라인 정의(YAML), 시크릿/환경 변수 설정.
- 로그 분석 도구 접근 권한(CloudWatch, ELK, Datadog 등).

## 상세 절차
1. **리허설 범위 정의**
   - 배포 후보 브랜치를 선택하고 파이프라인 트리거 방식을 확인한다(푸시, PR, 수동 실행 등).
   - 병렬/매트릭스 잡이 있는 경우 모두 실행되도록 설정한다.
2. **시크릿 및 환경 변수 검증**
   - 45단계 시크릿 주입 로직이 올바르게 작동하는지 확인하고 만료 여부를 점검한다.
   - 필요 시 임시 자격 증명을 발급해 테스트 후 폐기한다.
3. **파이프라인 실행**
   - 수동 트리거 또는 더미 커밋을 통해 전체 파이프라인을 실행한다.
   - 각 단계별 로그를 수집하고 실패 시 즉시 원인을 분류한다(코드 문제, 인프라 문제 등).
4. **품질 게이트 및 테스트 결과 검증**
   - SAST/SCA, 단위/통합 테스트 결과가 기준치를 충족하는지 확인한다.
   - 31단계 보안/품질 기준과 비교하여 누락된 검증 항목이 없는지 점검한다.
5. **아티팩트 검토**
   - 빌드 산출물(이미지, 패키지)이 저장소에 업로드되었는지 확인한다.
   - 해시/서명 검증을 48단계 업로드 검증 절차와 일치시키고 보고서를 갱신한다.
6. **개선 사항 반영 및 문서화**
   - 발견된 문제를 이슈로 등록하고 파이프라인 정의를 수정한다.
   - 실행 로그, 개선 사항을 95단계 배포 기록에 활용할 수 있도록 정리한다.

## 의사결정 포인트
- CI 실행을 정기적으로 리허설할지(예: 주간) 또는 주요 변경 시에만 할지.
- 실패 시 파이프라인을 중단할지, 부분 성공을 허용할지 정책 설정.
- 아티팩트 버전 규칙을 어떻게 적용할지(세맨틱 버전, 커밋 해시 등).

## 체크리스트
- [ ] 모든 CI 단계가 성공적으로 완료되었다.
- [ ] 품질 게이트/테스트 결과가 기준을 충족한다.
- [ ] 로그와 아티팩트가 보존 위치에 저장되었다.
- [ ] 개선 과제가 이슈로 등록되고 담당자가 지정되었다.

## 산출물 및 보관 위치
- CI 실행 로그 및 대시보드 스크린샷.
- 아티팩트 저장소 업로드 기록.
- 개선 과제 목록(`docs/reports/ci-rehearsal-action-items.md`).
