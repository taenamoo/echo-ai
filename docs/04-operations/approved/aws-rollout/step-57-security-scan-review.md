# 57단계. 보안 스캔 실행 및 리포트 검토

## 목적
- 31단계에서 정의한 보안 스캔 기준에 따라 애플리케이션과 인프라 아티팩트에 대한 스캔을 수행하고 결과를 분석한다.
- 발견된 취약점을 우선순위화하여 33단계 롤백/수정 정책과 연계한다.

## 선행조건
- [31단계 보안 스캔 및 품질 관리 기준](docs/04-operations/approved/aws-rollout/step-31-security-quality-baselines.md).
- [46단계 테스트 스위트 통합](docs/04-operations/approved/aws-rollout/step-46-test-suite-integration.md) 및 [56단계 빌드 테스트](docs/04-operations/approved/aws-rollout/step-56-initial-build-validation.md).
- 27~30단계 파이프라인 정의에서 스캔 도구 실행 위치가 지정되어 있을 것.

## 필요한 입력 자료
- SAST/SCA/컨테이너/IaC 스캔 도구 설정 파일 및 정책.
- 허용 가능한 위험 기준, 예외 승인 절차 문서.
- 취약점 티켓 템플릿, 트래킹 시스템(Jira 등) 접근 권한.

## 상세 절차
1. **스캔 환경 준비**
   - 빌드 아티팩트, 소스 코드, IaC 템플릿을 최신 상태로 체크아웃한다.
   - 도구 라이선스/토큰이 유효한지 확인한다.
2. **스캔 실행**
   - 정적 분석(SAST), 종속성 검사(SCA), 컨테이너 이미지 스캔, IaC 정책 검증을 실행한다.
   - 필요 시 43단계 CI 환경 또는 로컬에서 동일한 명령을 사용한다.
3. **결과 집계 및 우선순위화**
   - 취약점 심각도(CVSS 등)를 기준으로 분류하고 SLA를 적용한다.
   - 허용된 예외가 있는 경우 34단계 승인 절차에 따라 검증한다.
4. **티켓 및 수정 계획 작성**
   - 각 취약점에 대해 책임자, 수정 기한, 대응 전략(패치, 설정 변경 등)을 지정한다.
   - 51단계 구성 관리 또는 52단계 설정 변경이 필요한 경우 연계한다.
5. **리포트 배포 및 커뮤니케이션**
   - 결과 요약과 조치 계획을 이해관계자에게 공유한다.
   - 90단계 KPI에 보안 관련 지표를 추가한다.
6. **CI/CD 게이트 업데이트**
   - 임계값 초과 시 파이프라인이 실패하도록 품질 게이트를 조정한다.
   - 반복되는 취약점에 대한 교육/자동화 계획을 수립한다.

## 의사결정 포인트
- 스캔 실패 시 배포를 중단할지, 조건부 진행할지 정책.
- 취약점 예외 승인 기준과 만료 주기.
- 리포트 보관 위치 및 접근 권한.

## 체크리스트
- [ ] 모든 지정된 스캔이 최신 아티팩트에 대해 실행되었다.
- [ ] 취약점이 분류되고 티켓이 생성되었다.
- [ ] 품질 게이트가 최신 정책으로 업데이트되었다.
- [ ] 리포트와 조치 계획이 공유되었다.

## 산출물 및 보관 위치
- 스캔 결과 리포트 (`docs/reports/security-scans/<date>.md` 등).
- 취약점 트래킹 티켓 및 대시보드.
- 업데이트된 품질 게이트 설정 (`.github/workflows/`, `sonar-project.properties` 등).
