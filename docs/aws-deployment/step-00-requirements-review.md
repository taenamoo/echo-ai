# 0단계. 요구사항 명세 검토

## 목적
- AWS 배포 대상 서비스의 기능/비기능 요구사항을 명확하게 이해하여 이후 모든 단계의 기준점으로 활용한다.
- 요구사항 추적 매트릭스를 마련하여 각 단계에서 산출물을 검증할 수 있도록 한다.

## 선행조건
- 프로젝트 킥오프가 완료되어 기본 비즈니스 목표와 이해관계자가 정의되어 있어야 한다.
- 최신 제품 기획 문서(기능 명세, UX 흐름 등)가 접근 가능한 저장소에 존재해야 한다.

## 필요한 입력 자료
- 제품 기획서, 사용자 스토리, 서비스 수준 목표(SLO), 규제/보안 요구사항 등 모든 요구사항 문서.
- 기존 운영 중인 시스템이 있다면 현행 시스템 분석 자료.

## 상세 절차
1. **요구사항 문서 수집**
   - 기획, 운영, 보안 팀에서 관리하는 문서를 모두 수집한다.
   - 문서 저장 위치(Confluence, Google Drive 등)를 확인하고 접근 권한을 확보한다.
2. **요구사항 분류 및 정리**
   - 기능 요구사항, 비기능 요구사항(보안, 성능, 가용성 등)으로 구분한다.
   - 우선순위를 부여하고, 차후 반복 검토를 위한 버전 정보를 기록한다.
3. **추가 정보 확인**
   - 모호하거나 불완전한 요구사항은 담당 이해관계자에게 질의하여 명확히 한다.
   - 회의가 필요하면 의제를 준비하고 회의록 양식을 미리 작성한다.
4. **요구사항 추적 매트릭스(RTM) 작성**
   - 스프레드시트 또는 요구사항 관리 도구를 사용하여 항목 ID, 설명, 담당자, 관련 시스템을 정리한다.
   - 각 요구사항이 향후 어느 단계에서 해결될지(예: 네트워크, 보안, 애플리케이션) 태그를 부여한다.
5. **검토 회의 진행 및 승인**
   - 정리된 요구사항 목록을 이해관계자에게 공유하고 승인 서명을 받는다.
   - 회의록을 작성하여 결론과 미해결 항목을 명시한다.

## 의사결정 포인트
- 요구사항 변경 관리 프로세스(승인 권한, 변경 주기)를 설정해야 한다면 이해관계자와 합의한다.

## 체크리스트
- [ ] 모든 기능/비기능 요구사항 문서를 확보하였다.
- [ ] 모호한 요구사항에 대해 Q&A 및 회의록이 남아 있다.
- [ ] 요구사항 추적 매트릭스 초안이 작성되었고 버전이 부여되었다.
- [ ] 이해관계자의 승인 또는 확인 서명이 확보되었다.

## 산출물 및 보관 위치
- 요구사항 추적 매트릭스 (예: `docs/requirements/rtm.xlsx`).
- 회의록 및 Q&A 문서 (예: Confluence 페이지 혹은 팀 공유 드라이브).
