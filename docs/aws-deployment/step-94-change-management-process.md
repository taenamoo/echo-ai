# 94단계. 감사 추적 및 변경 관리 프로세스 확립

## 목적
- 배포, 인프라 변경, 운영 자동화 수정 등의 활동을 체계적으로 관리하고 감사 가능하도록 한다.
- 34단계 승인 절차와 81단계 보안 감사 요구사항을 통합해 추적성과 책임성을 확보한다.

## 선행조건
- [34단계 배포 승인 절차](docs/aws-deployment/step-34-deployment-approvals.md)가 정의되어 있어야 한다.
- [83단계 운영 대시보드](docs/aws-deployment/step-83-operations-dashboard.md)와 [90단계 KPI 추적](docs/aws-deployment/step-90-operations-kpi-definition.md)이 활성화되어 있어야 한다.
- 변경 요청을 기록할 도구(Jira, ServiceNow, GitHub Issues 등)와 권한 구성이 완료되어야 한다.

## 필요한 입력 자료
- 기존 변경 관리 정책 또는 ITSM 프로세스 문서.
- 변경 유형(표준/긴급/일반) 정의와 승인자 목록.
- 감사 요구사항(76단계)과 규제 준수 기준(7단계).

## 상세 절차
1. **프로세스 범위 및 역할 정의**
   - 애플리케이션 코드, 인프라, 보안 정책, 자동화 스크립트 등 변경 범위를 명확히 한다.
   - 변경 요청자, 검토자, 승인자, 배포 담당자 역할을 정의한다.
2. **워크플로우 설계 및 도구 구성**
   - 제출→평가→승인→배포→검증→종료 단계를 설계하고 상태 전이를 정의한다.
   - 도구에 템플릿, 필수 필드(영향도, 롤백 계획, 테스트 결과)를 설정한다.
3. **표준 변경 템플릿과 체크리스트 개발**
   - 반복되는 변경(패치, 설정 수정)은 표준 변경으로 분류하고 사전 승인 조건을 기록한다.
   - 긴급 변경 절차에는 사고 보고(96단계)와 연동되는 승인 절차를 포함한다.
4. **CI/CD 및 감사 로그 통합**
   - PR/배포 파이프라인에서 변경 요청 ID를 필수로 참조하게 구성한다.
   - CloudTrail, Config, Git 로그가 변경 요청과 매핑되도록 자동 태깅/웹훅을 설정한다.
5. **교육 및 시행**
   - 91·92단계 교육 세션에서 변경 프로세스를 소개한다.
   - 시범 운영 기간을 두고 개선 의견을 수집해 프로세스를 보완한다.

## 의사결정 포인트
- 표준/긴급 변경의 기준과 승인 레벨을 어떻게 정의할지.
- 변경 도구를 기존 ITSM과 통합할지, 별도 레포지토리를 운영할지 여부.
- 배포 자동화와 승인 연동을 수동 확인으로 둘지, 파이프라인에서 강제할지 결정.

## 체크리스트
- [ ] 변경 유형과 역할, 승인 절차가 문서화되었다.
- [ ] 변경 관리 도구가 설정되고 템플릿/필수 필드가 구성되었다.
- [ ] CI/CD 및 감사 로그와 변경 요청 ID가 연동되었다.
- [ ] 교육 및 시범 운영을 통해 사용자 피드백이 반영되었다.

## 산출물 및 보관 위치
- 변경 관리 정책 문서(`policies/change-management.md`).
- 변경 요청 템플릿(`itSM/templates/change-request-template.yaml`).
- 변경 로그 및 감사 보고(`logs/change-management/YYMM/`).
