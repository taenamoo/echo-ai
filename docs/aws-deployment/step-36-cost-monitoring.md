# 36단계. AWS 비용 모니터링 설정 초안

## 목적
- 개발 환경 운영 중 비용 초과를 예방하기 위해 모니터링 및 경보 체계를 설계한다.
- 비용 데이터 수집, 분석, 알림 절차를 정의하여 예산(6단계)을 준수한다.

## 선행조건
- 6단계 예산 및 비용 한도 설정 결과.
- 20단계 컴퓨팅 선택, 23~25단계 데이터 서비스 계획.
- 26단계 모니터링 프레임워크 결정.

## 필요한 입력 자료
- 월별/분기별 예산, 서비스별 비용 한도.
- 태깅 규칙 초안(8단계) 및 계정 구조(11단계).
- 사용량 패턴 추정치와 예약 인스턴스/절감 플랜 계획.

## 상세 절차
1. **비용 관리 목표 정의**
   - 개발 환경에서 허용 가능한 일/월 단위 비용 한도를 설정한다.
   - 예산 초과 경고, 차단 정책, 보고 주기를 명시한다.
2. **데이터 수집 구성**
   - AWS Cost Explorer, AWS Budgets, Cost and Usage Report(CUR) 활성화를 계획한다.
   - 계정/OU별 통합 청구를 사용하는 경우 수집 주기와 S3 저장소 구성을 정의한다.
3. **모니터링 및 대시보드 설계**
   - 26단계에서 결정한 모니터링 도구와 연동해 비용 대시보드를 구성한다.
   - 태그/계정/서비스별 뷰, 예산 대비 추이, 예약 인스턴스 활용도를 시각화한다.
4. **알림 및 에스컬레이션 정책 작성**
   - AWS Budgets 알림 임계값(예: 50%, 80%, 100%)을 설정한다.
   - 알림 채널(Slack, 이메일, ITSM)과 대응 책임자를 지정한다.
5. **최적화 프로세스 정의**
   - 주기적 비용 리뷰 회의, 절감 기회 식별, 권장 조치 실행 절차를 마련한다.
   - Trusted Advisor, Compute Optimizer, 비용 탐지 스크립트 활용 방안을 문서화한다.
6. **증적 및 보고 체계 준비**
   - 재무팀 공유용 리포트 템플릿과 저장 위치를 정의한다.
   - 감사 시 요구되는 기록 보관 기간 및 접근 제어 정책을 명시한다.

## 의사결정 포인트
- CUR 저장 주기와 보관 기간.
- 비용 초과 시 자동 차단 여부와 책임 조직.

## 체크리스트
- [ ] 비용 모니터링 목표 및 알림 임계값이 정의되었다.
- [ ] Cost Explorer/Budgets/CUR 구성 계획이 수립되었다.
- [ ] 대시보드 설계와 알림 채널이 문서화되었다.
- [ ] 비용 최적화 리뷰 및 보고 절차가 마련되었다.
