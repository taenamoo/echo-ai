# 27단계. CI/CD 파이프라인 요구사항 정의

## 목적
- 소스 코드 변경이 빌드·테스트·배포 단계로 흐르는 전체 파이프라인 요구사항을 명확히 문서화한다.
- 품질, 보안, 승인 절차를 자동화하기 위한 기준선을 마련한다.

## 선행조건
- 3단계 버전 전략, 4단계 IaC 결정, 5단계 환경 정의.
- 20~26단계 인프라 및 운영 설계 결과.
- 21단계 아티팩트 전략, 22단계 시크릿 관리 방안.

## 필요한 입력 자료
- 코드 레포지토리 구조, 브랜치 보호 규칙, 코드 리뷰 정책.
- 테스트 종류(Unit/Integration/E2E), 커버리지 목표, 품질 게이트.
- 컴플라이언스 요구사항(감사 로그, 승인 기록, 변경 관리).

## 상세 절차
1. **파이프라인 스테이지 정의**
   - 소스, 빌드, 테스트, 보안 스캔, 아티팩트 발행, 배포, 검증 단계로 구분한다.
   - 각 단계의 입력/출력, 실패 시 처리 방식을 정의한다.
2. **환경 및 브랜치 매핑**
   - 브랜치/태그별로 어떤 환경(개발, 스테이징, 운영)에 배포할지 결정한다.
   - 수동 승인 게이트(34단계), 배포 전략(32단계)과 연계한다.
3. **품질 및 보안 통제 정의**
   - 테스트 커버리지 기준, 코드 스멜/린트 정책, 취약점 스캔 요구사항을 명시한다.
   - 감사 로그 보존, 변경 이력 추적 방식을 정한다.
4. **인프라/데이터 작업 포함 여부 결정**
   - IaC, 데이터베이스 마이그레이션, 캐시 시드 작업을 어느 단계에서 수행할지 결정한다.
   - 롤백 절차(33단계)와 동기화한다.
5. **알림 및 관찰성 요구사항 정의**
   - 파이프라인 실패/성공 알림 채널(Slack, 이메일, PagerDuty)을 설정한다.
   - 26단계 모니터링 도구와 통합 포인트를 문서화한다.
6. **규정 준수 및 감사 요구사항 명시**
   - 변경 승인자, 릴리스 노트, 배포 증적 자료를 어떻게 수집할지 정의한다.
   - 규제 감사 대비를 위해 로그/증적 보존 기간을 설정한다.

## 의사결정 포인트
- 단일 파이프라인 vs. 서비스별 파이프라인 구성.
- 인프라와 애플리케이션 배포를 분리할지 통합할지 여부.

## 체크리스트
- [ ] 파이프라인 단계와 입력/출력이 정의되었다.
- [ ] 품질, 보안, 감사 요구사항이 문서화되었다.
- [ ] 환경/브랜치 매핑과 승인 흐름이 정해졌다.
- [ ] 관찰성 및 알림 요구사항이 수립되었다.
