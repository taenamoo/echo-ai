# 95단계. 릴리즈 노트 및 배포 기록 관리

## 목적
- 배포 내역과 주요 변경 사항을 일관되게 기록하여 회귀 분석과 커뮤니케이션을 원활히 한다.
- KPI(90단계) 추적, 포스트모템(96단계), 피드백 루프(99단계)에 필요한 데이터 소스를 제공한다.

## 선행조건
- [94단계 변경 관리 프로세스](docs/aws-deployment/step-94-change-management-process.md)가 시행 중이어야 한다.
- CI/CD 파이프라인(43~49단계)에서 릴리즈 메타데이터를 수집할 수 있어야 한다.
- 서비스 버저닝 전략(3단계)이 확정되어 있어야 한다.

## 필요한 입력 자료
- 배포 일정, 변경 요청 ID, 커밋/태그 정보.
- 테스트 결과 요약(67~69단계), 보안 검토 상태(70·81단계).
- 이해관계자 공지 채널 및 배포 승인 기록.

## 상세 절차
1. **릴리즈 노트 템플릿 정의**
   - 버전, 배포 일시, 환경, 담당자, 변경 요약, 영향, 롤백 계획, 관련 티켓 필드를 포함한다.
   - 34단계 승인 정보와 94단계 변경 요청 ID를 연계할 수 있도록 필드를 설정한다.
2. **데이터 수집 자동화**
   - CI/CD 파이프라인에서 커밋 메시지, 테스트 결과, 아티팩트 정보를 템플릿에 자동 주입한다.
   - 수동 입력이 필요한 항목(위험/영향 분석)은 검토자가 작성하도록 지정한다.
3. **검토 및 승인**
   - 배포 전 변경 검토 회의에서 릴리즈 노트를 확인하고 승인받는다.
   - 긴급 변경의 경우 96단계 포스트모템을 예약하고 릴리즈 노트에 명시한다.
4. **배포 후 업데이트 및 배포**
   - 실제 배포 결과(성공/실패, 소요 시간, 모니터링 이슈)를 기록한다.
   - 이해관계자에게 릴리즈 요약을 공지하고 83단계 대시보드에 링크를 게시한다.
5. **보관 및 검색 체계 구축**
   - 릴리즈 노트를 버전 관리 저장소나 위키에 보관하고 검색 태그를 설정한다.
   - 98단계 로드맵 업데이트 시 참고할 수 있도록 연간/분기별 요약 리포트를 생성한다.

## 의사결정 포인트
- 릴리즈 노트를 PR 머지 시 생성할지, 배포 파이프라인 완료 후 생성할지 결정.
- 공개 범위(전체 팀 vs. 제한된 이해관계자)와 보관 기간 정의.
- 실패한 배포도 동일한 템플릿으로 기록할지 여부.

## 체크리스트
- [ ] 릴리즈 노트 템플릿이 정의되고 변경 관리 프로세스와 연동되었다.
- [ ] CI/CD 파이프라인에서 필요한 메타데이터가 자동 수집된다.
- [ ] 배포 직후 릴리즈 노트가 업데이트되고 이해관계자에게 공유된다.
- [ ] 릴리즈 기록이 검색 가능하게 보관되고 정기 리포트로 요약된다.

## 산출물 및 보관 위치
- 릴리즈 노트 저장소(`docs/releases/YYYY/Release-<version>.md`).
- 릴리즈 요약 대시보드(`dashboards/releases/status.json`).
- 분기별 릴리즈 리포트(`reports/releases/2024Qx-summary.pdf`).
