# 41단계. IAM 역할/사용자 생성 및 검증

## 목적
- 12단계에서 설계한 IAM 역할과 정책을 실제 AWS 계정에 생성하여 운영에 필요한 접근 제어를 구축한다.
- 생성된 역할·사용자가 CloudTrail, Config 등 감사 로그에 기록되도록 검증하고 운영 표준을 마련한다.

## 선행조건
- [12단계 IAM 설계 문서](docs/aws-deployment/step-12-iam-design.md).
- [13단계 SSO/자격 증명 전략](docs/aws-deployment/step-13-sso-identity-management.md).
- 8단계 거버넌스 원칙과 17단계 네트워크 보안 정책.

## 필요한 입력 자료
- 역할/사용자 요구사항 목록(RBAC 매트릭스, RACI).
- 사용할 IAM 정책(JSON) 정의 또는 템플릿.
- AWS 계정 ID, 조직(OUs) 및 사용할 프로비저닝 도구(IaC, AWS CLI 등) 정보.

## 상세 절차
1. **역할/사용자 인벤토리 확정**
   - 운영자, 배포 파이프라인, 애플리케이션 서비스 역할 등 유형별 목록을 작성한다.
   - 1단계 RACI와 34단계 승인 절차에서 요구하는 권한을 매핑한다.
2. **정책 템플릿 검토 및 적용 범위 정의**
   - 최소 권한 원칙에 따라 관리형 정책 vs. 커스텀 정책 사용 여부를 결정한다.
   - 8단계 거버넌스 문서의 태깅/명명 규칙을 반영해 정책 이름을 확정한다.
3. **프로비저닝 방식 선택**
   - IaC(CloudFormation/Terraform/CDK) 또는 AWS IAM Identity Center 프로비저닝을 선택한다.
   - 선택한 방식에 따라 코드/스크립트를 작성하고 형상 관리 저장소에 커밋한다.
4. **역할·사용자 생성**
   - 각 역할에 필요한 신뢰 정책(assume role)을 설정한다.
   - 세션 태그, 조건부 MFA, 시간 제한 등 추가 보안 옵션을 적용한다.
5. **감사 및 로깅 설정 검증**
   - CloudTrail, AWS Config, Access Analyzer에서 변경 사항이 기록되는지 확인한다.
   - IAM Access Advisor/Access Analyzer를 사용하여 과도한 권한이 없는지 검토한다.
6. **접근 테스트 및 문서화**
   - 수동 또는 자동 스크립트로 주요 API 호출을 테스트하고 성공/실패 로그를 수집한다.
   - 문제 발생 시 12단계 설계 문서를 업데이트하고 재검토한다.

## 의사결정 포인트
- 수동 콘솔 생성 vs. IaC 자동화 여부.
- 역할 체인(assume role) 구조 및 세션 지속 시간 설정.
- SSO를 통한 프로비저닝 범위와 운영 계정 분리 수준.

## 체크리스트
- [ ] 역할/사용자 목록이 최신 설계 문서와 일치한다.
- [ ] 정책이 최소 권한 원칙을 충족하며 검토 로그가 저장되었다.
- [ ] CloudTrail/Config에서 변경 내역을 확인했다.
- [ ] 접근 테스트 결과가 문서화되어 있다.

## 산출물 및 보관 위치
- IAM 프로비저닝 코드/스크립트 (`infra/iam/` 또는 `docs/aws-deployment/iam-provisioning/`).
- 접근 테스트 보고서 (`docs/reports/iam-access-validation.md`).
- 감사 로그 위치 및 보존 정책 문서 (`docs/runbooks/security-logging.md`).
