# 3단계. 버전 관리 전략 확정

## 목적
- 프로젝트 전체에 적용할 Git 브랜치 전략과 릴리스 프로세스를 정의한다.
- 이후 CI/CD 구성 및 협업 규칙의 기준을 제공한다.

## 선행조건
- 1단계에서 확정된 이해관계자 및 승인자 목록.
- 2단계에서 정리한 애플리케이션 구성 요소와 저장소 구조 후보.

## 필요한 입력 자료
- 현재 사용 중인 리포지토리 현황, 브랜치 정책, 코드 리뷰 프로세스.
- 팀 규모, 동시 개발 스트림 수, 릴리스 빈도에 대한 요구사항.

## 상세 절차
1. **현재 버전 관리 방식 진단**
   - 기존 프로젝트에서 사용 중인 브랜치 전략과 릴리스 주기를 조사한다.
   - 문제점(충돌 빈도, 배포 지연 등)을 기록한다.
2. **전략 후보 도출**
   - Git Flow, GitHub Flow, Trunk-based 등 후보 전략을 비교 표로 정리한다.
   - 각 전략이 CI/CD, QA, 배포 승인에 미치는 영향을 평가한다.
3. **저장소 구조 결정**
   - 단일 모노레포 vs 다중 리포지토리 여부를 결정한다.
   - 2단계에서 정리한 컴포넌트별 소유팀을 고려하여 권한 모델을 정의한다.
4. **브랜치/태그 규칙 수립**
   - 브랜치 명명 규칙(feature/, hotfix/, release/ 등)과 보호 규칙(PR 리뷰, CI 통과 필수)을 설정한다.
   - 릴리스 태그 규칙(예: `v<MAJOR>.<MINOR>.<PATCH>`)과 체인지로그 작성 방식을 결정한다.
5. **릴리스 및 롤백 절차 정의**
   - 프로덕션 배포 승인 흐름과 롤백 전략을 문서화한다.
   - 긴급 패치 시 사용할 프로세스(Hotfix 브랜치 등)를 포함한다.
6. **정책 문서화 및 승인**
   - 결정된 전략을 문서화하여 이해관계자 승인(Dev Lead, Ops Lead 등)을 받는다.
   - 개발자 온보딩 가이드에 전략 내용을 반영한다.

## 의사결정 포인트
- 모노레포 도입 여부 및 이를 관리하기 위한 도구(PNPM 워크스페이스 등) 사용 결정.
- 코드 리뷰 필수 인원 수, 자동 병합 허용 여부.

## 체크리스트
- [ ] 브랜치 전략과 릴리스 절차가 문서화되었다.
- [ ] 태그/버전 명명 규칙과 체인지로그 프로세스가 정의되었다.
- [ ] 접근 권한 및 보호 규칙이 설정되었다.
- [ ] 이해관계자의 서명 혹은 이메일 승인을 확보했다.

## 산출물 및 보관 위치
- 버전 관리 정책 문서 (예: `docs/governance/version-control-policy.md`).
- 브랜치 보호 규칙 설정 스크린샷 또는 IaC 코드.
