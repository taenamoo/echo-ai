# 12단계. IAM 사용자 및 역할 설계

## 목적
- 최소 권한 원칙에 따라 IAM 사용자, 역할, 정책을 설계하여 보안과 운영 효율을 확보한다.
- 계정 및 환경별 접근 제어 모델을 표준화한다.

## 선행조건
- 1단계 이해관계자 및 책임자 목록.
- 7단계 보안 및 규제 요구사항 정의서.
- 11단계에서 확정된 멀티 계정 및 OU 구조.

## 필요한 입력 자료
- 팀/서비스별 업무 시나리오와 필요한 AWS 서비스 목록.
- 인증 방식(SSO, MFA 등)과 감사 요구사항.
- 기존 온프레미스 또는 IdP 그룹 구조.

## 상세 절차
1. **역할 분류 정의**
   - 사용자 유형(개발자, 운영자, 감사자 등)을 구분하고 책임 범위를 명확히 한다.
   - 환경별(개발/스테이징/운영) 접근 수준을 매트릭스로 정리한다.
2. **정책 설계**
   - AWS 관리형 정책 사용 가능 여부를 검토하고, 필요 시 커스텀 정책을 설계한다.
   - 8단계 거버넌스 원칙에 맞춰 태그 기반 조건과 리소스 범위를 제한한다.
3. **역할-정책 매핑**
   - 역할(예: `Developer`, `Ops`, `Auditor`)마다 연결할 정책 세트를 정의한다.
   - 교차 계정 액세스가 필요하면 IAM 역할 신뢰 정책을 문서화한다.
4. **MFA 및 세션 관리 설정**
   - 필수 MFA 정책과 세션 타임아웃, 권한 상승 절차를 정의한다.
   - AWS CLI/SDK 사용 시 프로파일 구성 방법을 안내한다.
5. **프로비저닝 및 오프보딩 프로세스 수립**
   - 신규 사용자 생성, 권한 부여, 종료 시 권한 회수 절차를 정의한다.
   - 자동화 도구(예: Terraform, AWS SSO 프로비저닝) 사용 여부를 결정한다.
6. **감사 및 모니터링 계획**
   - IAM Access Analyzer, Credential Report 등을 활용한 정기 점검 방법을 명시한다.
   - 비정상 활동 탐지를 위한 CloudWatch/CloudTrail 경보 기준을 정의한다.
7. **승인 및 교육**
   - 설계 결과를 보안팀과 운영팀에 공유하고 승인을 획득한다.
   - 사용자 온보딩 가이드를 작성하여 배포한다.

## 의사결정 포인트
- 역할 기반 접근 제어(RBAC) vs. 속성 기반 정책(ABAC) 적용 범위.
- IaC 도구를 통한 정책 관리 자동화 수준.

## 체크리스트
- [ ] 역할 분류표 및 권한 매트릭스가 완성되었다.
- [ ] IAM 정책 JSON 및 신뢰 정책이 검토되었다.
- [ ] 프로비저닝/오프보딩 절차가 문서화되었다.
- [ ] MFA, 세션 관리, 감사 계획이 승인되었다.
