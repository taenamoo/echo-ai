# 5단계. 배포 환경 구분 정의

## 목적
- 개발(dev), 스테이징(stage), 운영(prod) 등 환경을 구분하고 격리 전략을 수립한다.
- 각 환경에 필요한 AWS 계정, VPC, 네트워크 정책을 정의하여 이후 설계를 위한 기반을 마련한다.

## 선행조건
- 2단계 아키텍처 자료 및 4단계 IaC 전략 결정 내용.
- 6단계 예산 수립에 반영해야 하므로 재무 담당자와 협의 가능한 상태여야 한다.

## 필요한 입력 자료
- 조직의 환경 분리 정책(보안 규정, 접근 통제 기준).
- 예상되는 테스트/배포 빈도 및 사용량.
- 멀티 계정 전략 필요성 관련 정보(AWS Organizations 정책 등).

## 상세 절차
1. **환경 정의 워크숍 진행**
   - 개발, QA, 운영, 보안 팀과 함께 환경별 목적과 사용 시나리오를 정의한다.
   - 필수 환경과 선택 환경(예: perf, sandbox)을 분류한다.
2. **격리 수준 결정**
   - 환경별 AWS 계정 분리 여부를 결정한다.
   - 동일 계정 내 VPC 분리 또는 네임스페이스 격리(EKS 등)를 사용할지 결정한다.
3. **네이밍 및 태깅 규칙 확정**
   - 계정, VPC, 자원 태그에 사용할 접두사/접미사 규칙을 작성한다.
   - 8단계 거버넌스 원칙과 연계하여 표준화한다.
4. **접근 제어 정책 정의**
   - IAM 역할, SSO 그룹, 권한 경계를 환경별로 정의한다.
   - 배포 도구(GitHub Actions, Jenkins 등)가 사용할 역할을 명시한다.
5. **운영 및 데이터 분리 전략 수립**
   - 환경 간 데이터 격리(예: 별도 데이터베이스, S3 버킷) 방식을 설계한다.
   - 환경 간 테스트 데이터 동기화 절차와 마스킹 정책을 정의한다.
6. **문서화 및 승인**
   - 환경 정의 문서를 작성하여 이해관계자(보안, 운영, 재무) 승인서를 확보한다.
   - IaC 템플릿 혹은 계정/VPC 생성 절차서를 첨부한다.

## 의사결정 포인트
- AWS Control Tower/Organizations를 통한 계정 자동화 도입 여부.
- 네트워크 레벨 격리 수준(VPC Peering, Transit Gateway 필요 여부).

## 체크리스트
- [ ] 환경별 목적, 사용 시나리오, 자원 범위가 명확히 정의되었다.
- [ ] 계정/VPC/네트워크 격리 전략이 결정되었다.
- [ ] 접근 제어 정책과 태깅 규칙이 문서화되었다.
- [ ] 이해관계자 승인(운영, 보안, 재무)을 확보했다.

## 산출물 및 보관 위치
- 환경 정의서 (예: `docs/environment/environment-definition.md`).
- 계정/VPC 생성 IaC 템플릿 또는 절차서.
