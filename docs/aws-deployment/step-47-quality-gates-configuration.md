# 47단계. 코드 품질 게이트 구성

## 목적
- 31단계에서 정의한 품질 기준을 CI 파이프라인과 코드 저장소에 적용하여 기준 미달 시 배포가 중단되도록 한다.
- 테스트/스캔 결과를 승인 프로세스(34단계)와 연결해 자동화된 품질 관리 체계를 구축한다.

## 선행조건
- [31단계 보안 스캔 및 품질 관리 기준 확립](docs/aws-deployment/step-31-security-quality-baselines.md).
- 43~46단계에서 구성한 CI 파이프라인 및 테스트 통합.
- 34단계 배포 승인 절차 문서.

## 필요한 입력 자료
- 품질 게이트 임계값(취약점 심각도, 커버리지 %, 린트 규칙 등).
- 스캔 도구/테스트 프레임워크 결과 파일 경로.
- 승인 워크플로우 구성 정보(예: GitHub 보호 규칙, Jenkins quality gate 플러그인).

## 상세 절차
1. **품질 기준 매핑**
   - 31단계 문서의 기준을 테스트/스캔 결과 지표와 매핑한다.
   - 필수/선택 기준을 구분하고 예외 승인 절차를 정의한다.
2. **도구 통합 구성**
   - SonarQube, CodeGuru, Inspector 등의 결과를 CI 파이프라인에서 평가하도록 설정한다.
   - 품질 게이트를 통과하지 못하면 파이프라인이 실패하도록 조건을 추가한다.
3. **저장소 보호 규칙 적용**
   - Pull Request 병합 전에 품질 체크가 완료되어야 하도록 브랜치 보호 규칙을 설정한다.
   - 필수 상태 체크, 리뷰어 수, 승인자 조건을 1/34단계 문서와 일치시킨다.
4. **알림 및 보고**
   - 품질 게이트 실패 시 해당 팀 채널로 알림을 전송하고 자동 이슈를 생성한다.
   - 주간/월간 품질 리포트를 생성하여 73단계 문서화에 반영한다.
5. **예외 관리 절차 수립**
   - 긴급 배포 시 예외 승인 절차와 승인자 역할을 정의한다.
   - 예외 케이스를 추적하고 정기적으로 재검토한다.
6. **검증 및 튜닝**
   - 샘플 실패 시나리오를 실행해 게이트가 정확히 차단하는지 확인한다.
   - 거짓 양성/음성 사례를 분석하여 규칙을 조정한다.

## 의사결정 포인트
- 품질 게이트 임계값을 환경별로 다르게 적용할지 여부.
- 예외 승인 권한과 기간 제한 설정.
- 품질 지표를 KPI(90단계)로 포함할지 여부.

## 체크리스트
- [ ] 품질 게이트 규칙이 파이프라인과 저장소에 적용되었다.
- [ ] 실패 시 알림 및 이슈 자동 생성이 작동한다.
- [ ] 예외 승인 절차가 문서화되고 책임자가 지정되었다.
- [ ] 품질 리포트가 정기적으로 생성되고 보관된다.

## 산출물 및 보관 위치
- 품질 게이트 구성 파일 (`sonar-project.properties`, `ci/pipelines/quality-gates.yaml`).
- 브랜치 보호 및 승인 정책 문서 (`docs/policies/repository-quality.md`).
- 품질 지표 대시보드 또는 보고서 (`docs/reports/quality-metrics.md`).
