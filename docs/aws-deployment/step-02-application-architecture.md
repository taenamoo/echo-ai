# 2단계. 배포 대상 애플리케이션 구조 파악

## 목적
- AWS 배포 대상 서비스의 기술 스택, 구성 요소, 아키텍처 의존성을 완전하게 파악한다.
- 이후 인프라 설계와 배포 전략 수립을 위한 기초 자료를 확보한다.

## 선행조건
- 0단계 요구사항 추적 매트릭스.
- 1단계에서 확정한 이해관계자 및 담당자 목록.

## 필요한 입력 자료
- 현행 시스템 아키텍처 다이어그램, 서비스 토폴로지.
- 소스 저장소 정보, 빌드 도구, 런타임 환경 문서.
- 데이터베이스 스키마, 통합 시스템 API 명세.

## 상세 절차
1. **애플리케이션 구성 요소 식별**
   - 서비스가 모놀리식인지 마이크로서비스인지 구분한다.
   - 프런트엔드, 백엔드, 배치, 데이터 파이프라인 등 역할별 컴포넌트를 목록화한다.
2. **기술 스택 정리**
   - 사용 언어, 프레임워크, 런타임 버전, 패키지 관리 도구를 정리한다.
   - 각 컴포넌트의 빌드 및 실행 방식(예: Node.js + pnpm, Python + Poetry)을 명시한다.
3. **의존성 및 통합 포인트 파악**
   - 외부 API, 내부 서비스, 메시지 큐, 데이터베이스 등 연결 지점을 도식화한다.
   - 네트워크 요구사항(포트, 프로토콜, 인증 방식)을 기록한다.
4. **아키텍처 다이어그램 최신화**
   - Lucidchart, Draw.io 등 도구를 사용하여 현재 구조를 시각화한다.
   - 1단계 이해관계자와 검토하여 최신 상태로 확정한다.
5. **제약 조건 확인**
   - 라이선스, 온프레미스 종속성, 데이터 거주 요구사항 등 AWS 마이그레이션에 영향을 주는 제약을 파악한다.
   - 마이그레이션 시 리팩터링이 필요한 영역을 식별한다.
6. **문서화 및 공유**
   - 정리한 정보를 문서(예: `docs/architecture/current-system.md`)로 정리하고 버전 관리한다.
   - 관련 이해관계자에게 공유하고 피드백을 반영한다.

## 의사결정 포인트
- 리팩터링이 필요한 컴포넌트에 대해 기술 부채 해결 전략을 수립할지 여부.
- 온프레미스 연동을 유지할지, AWS 서비스로 대체할지 결정해야 한다면 추가 워크숍을 계획한다.

## 체크리스트
- [ ] 모든 컴포넌트의 기술 스택과 런타임 정보가 정리되었다.
- [ ] 의존성/통합 포인트에 대한 상세 문서와 다이어그램이 작성되었다.
- [ ] 아키텍처 다이어그램이 최신 상태로 승인되었다.
- [ ] AWS 마이그레이션 시 제약 조건과 고려사항이 문서화되었다.

## 산출물 및 보관 위치

- 제약 사항 목록 및 리스크 로그.
