# 19단계. 네트워크 연결성 테스트 계획 수립

## 목적
- 설계된 네트워크 토폴로지와 보안 정책이 실제로 의도한 대로 동작하는지 검증하기 위한 테스트 계획을 수립한다.
- 연결성 문제를 조기에 발견하여 배포 일정에 영향을 주지 않도록 한다.

## 선행조건
- 14~18단계 네트워크/보안/DNS 설계 문서.
- 16단계 VPC 생성 계획과 17단계 보안 정책.
- 테스트 환경 또는 PoC 계정 준비 상태.

## 필요한 입력 자료
- 테스트 대상 리소스 목록(서브넷, 인스턴스, 게이트웨이, DNS 레코드 등).
- 성공/실패 기준, 지표, 경보 요구사항.
- 테스트 도구 목록(aws reachability analyzer, ping, traceroute, curl 등).

## 상세 절차
1. **테스트 범위 정의**
   - 내부 서비스 간, 외부 인터넷, 온프레미스 연결 등 필요한 시나리오를 정의한다.
   - 네트워크 경로별로 기대 결과(허용/차단)를 명시한다.
2. **테스트 환경 준비**
   - 임시 인스턴스 또는 컨테이너를 배포하여 테스트 포인트를 구성한다.
   - 필요한 보안 그룹, IAM 권한, 도구 설치를 수행한다.
3. **테스트 케이스 설계**
   - 각 시나리오에 대한 구체적 단계(명령어, 예상 결과)를 문서화한다.
   - DNS 해상도, 라우팅, 포트 접근성, 지연 시간 측정 등을 포함한다.
4. **자동화 및 스케줄링 계획**
   - CI/CD 또는 IaC 파이프라인에 테스트 자동화를 통합할지 여부를 결정한다.
   - 정기적 회귀 테스트 주기와 보고 체계를 정의한다.
5. **이슈 대응 및 롤백 전략**
   - 테스트 실패 시 원인 분석 절차와 책임자를 지정한다.
   - 긴급 변경/예외 처리 절차를 문서화한다.
6. **결과 기록 및 보고 체계**
   - 테스트 결과 템플릿을 정의하고 저장 위치를 결정한다.
   - 이해관계자에게 공유할 요약 보고 형식을 설계한다.
7. **승인 및 일정 확정**
   - 테스트 계획을 검토받아 승인하고 실행 일정을 확정한다.

## 의사결정 포인트
- 자동화 도구(AWS Reachability Analyzer, 네트워크 시뮬레이터) 도입 여부.
- 테스트 주기(사전 배포, 정기 회귀)의 범위와 책임자.

## 체크리스트
- [ ] 테스트 범위 및 시나리오 목록이 완성되었다.
- [ ] 테스트 케이스와 자동화 계획이 문서화되었다.
- [ ] 실패 대응 절차와 보고 체계가 정의되었다.
- [ ] 승인 및 일정이 확정되었다.

## 산출물 및 보관 위치
- 네트워크 테스트 계획서 (`docs/network/connectivity-test-plan.md`).
- 자동화 스크립트 또는 파이프라인 설정 (`infra/network/tests/`).
- 테스트 결과 보고 템플릿 (`docs/network/reports/connectivity-report-template.md`).
