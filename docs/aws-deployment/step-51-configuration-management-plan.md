# 51단계. 구성 관리(Ansible, SSM 등) 플랜 수립

## 목적
- 애플리케이션과 운영체제 수준의 설정을 일관되게 배포하기 위한 구성 관리 전략을 수립한다.
- 50단계에서 작성한 IaC 템플릿과 연계하여 인스턴스/컨테이너가 초기화될 때 표준 구성을 자동 적용하도록 한다.

## 선행조건
- [41단계 IAM 프로비저닝](docs/aws-deployment/step-41-iam-provisioning.md)으로 필요한 역할과 권한이 준비되어 있을 것.
- [50단계 IaC 템플릿](docs/aws-deployment/step-50-iac-deployment-templates.md)과 환경별 변수 정의가 완료되어 있을 것.
- 7·8단계에서 확정한 보안 및 거버넌스 정책.

## 필요한 입력 자료
- 서버/컨테이너 인벤토리와 운영체제 이미지 목록.
- 필수 패키지, 서비스, 설정 파일 요구사항(예: `/etc/` 템플릿, 애플리케이션 런타임 옵션).
- 변경 관리/승인 절차와 롤백 기준.

## 상세 절차
1. **구성 범위 및 레벨 정의**
   - OS 패키지, 런타임 언어, 에이전트(모니터링/백업), 애플리케이션 설정 등 관리 항목을 카테고리화한다.
   - AMI, 컨테이너 이미지, Lambda Layer 등 자산 유형별로 적용 범위를 구분한다.
2. **도구 및 실행 방식 확정**
   - Ansible, AWS Systems Manager(SSM), AWS OpsWorks, Chef 등 후보를 비교하고 4단계 IaC 결정과 일치시키는 도구를 선택한다.
   - SSM 사용 시 Session Manager, State Manager, Automation 문서 사용 여부를 정의한다.
3. **구성 템플릿/플레이북 설계**
   - 공통 역할과 환경별 역할을 분리하여 재사용성을 높인다.
   - 시크릿, 인증 정보는 22·38단계 시크릿 운영 정책에 따라 Parameter Store/Secrets Manager 참조로 정의한다.
4. **CI/CD 파이프라인 통합 계획**
   - 30단계 빌드 파이프라인, 49단계 CD 파이프라인에서 구성 관리 스크립트를 호출하는 시점을 정의한다.
   - 드리프트 탐지 및 재적용 주기(예: cron, EventBridge)를 설정한다.
5. **검증 및 롤백 전략 수립**
   - 테스트 환경에서 구성 적용 후 상태 검사 스크립트를 실행한다.
   - 실패 시 롤백 방법(이전 AMI 재배포, 이전 버전 플레이북 재실행 등)을 정의한다.
6. **문서화 및 운영 인계**
   - 실행 방법, 인자, 실패 대응 Runbook을 작성하여 74단계 운영 핸드오프 자료에 포함한다.
   - 구성 변경 시 변경 관리 시스템(Jira 등)과 연동하는 절차를 마련한다.

## 의사결정 포인트
- 에이전트 기반 vs. 에이전트리스 구성 관리 도구 선택.
- 구성 적용 트리거(배포 시점, 주기적 재적용 등)와 자동화 수준.
- 구성 항목의 버전 관리 저장소 구조(모놀리식 vs. 환경별 분리).

## 체크리스트
- [ ] 구성 항목 목록과 책임자가 문서화되었다.
- [ ] 도구 선택 및 접근 권한 구성이 완료되었다.
- [ ] 플레이북/Automation 문서가 저장소에 커밋되었다.
- [ ] 검증/롤백 절차가 테스트 환경에서 검증되었다.

- 운영 Runbook (`docs/runbooks/config-management.md`).
