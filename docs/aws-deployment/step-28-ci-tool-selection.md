# 28단계. CI 도구 선택

## 목적
- 정의된 파이프라인 요구사항을 충족할 수 있는 지속적 통합(CI) 도구를 선정한다.
- 코드 빌드, 테스트, 아티팩트 생성 자동화를 위한 플랫폼과 운영 모델을 결정한다.

## 선행조건
- 3단계 버전 전략, 4단계 IaC 결정.
- 21단계 아티팩트 전략, 22단계 시크릿 관리.
- 27단계 CI/CD 요구사항 정의.

## 필요한 입력 자료
- 조직의 기존 CI 도구 사용 현황, 라이선스 조건.
- 빌드 언어/프레임워크, 런타임 요구사항, 병렬 빌드 필요성.
- 네트워크 격리, 보안 정책, 감사 요구사항.

## 상세 절차
1. **후보 도구 목록 작성**
   - GitHub Actions, AWS CodeBuild, GitLab CI, Jenkins, CircleCI 등.
   - SaaS vs. 자체 호스팅 여부를 구분한다.
2. **기능 및 통합성 평가**
   - IaC, 컨테이너 빌드, 캐시, 매트릭 수집, 시크릿 통합 지원 여부를 확인한다.
   - SCM, 이슈 추적, 알림 시스템과의 통합성을 평가한다.
3. **보안 및 컴플라이언스 검토**
   - 실행 환경 격리, IAM 역할, 네트워크 접근 제어를 검토한다.
   - 감사 로그, 권한 분리, 시크릿 보호 기능을 확인한다.
4. **비용 및 운영 모델 분석**
   - 빌드 분당 비용, 러너 인프라 비용, 라이선스 요금을 계산한다.
   - 운영 인력 필요 여부, SLA, 지원 정책을 검토한다.
5. **PoC 및 벤치마크 수행**
   - 대표 서비스에 대한 샘플 파이프라인을 실행하여 성능과 안정성을 평가한다.
   - 장애 및 재시도 시나리오, 캐시 효율을 테스트한다.
6. **선정 및 표준 문서화**
   - 최종 도구를 결정하고 선택 기준, 비교 결과, 제약사항을 문서화한다.
   - 운영 책임자, 러너 관리 절차, 확장 전략을 정의한다.

## 의사결정 포인트
- SaaS 기반 도구 사용 가능 여부(데이터 주권, 네트워크 제약).
- 자체 러너/빌더를 운영할지 여부.

## 체크리스트
- [ ] 후보 도구 비교 평가표가 작성되었다.
- [ ] 보안, 비용, 운영 요구사항 검토가 완료되었다.
- [ ] PoC 결과와 최종 결정이 승인되었다.
- [ ] 운영 모델과 책임자가 정의되었다.
