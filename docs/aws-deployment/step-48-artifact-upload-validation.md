# 48단계. 아티팩트 저장소 업로드 검증

## 목적
- CI 파이프라인에서 생성된 빌드 산출물이 42단계에서 구축한 아티팩트 저장소(S3, ECR 등)에 안전하게 업로드되고 검증되도록 한다.
- 업로드된 아티팩트의 무결성, 버전 관리, 접근 제어가 정책을 준수하는지 확인한다.

## 선행조건
- [42단계 S3/ECR 등 핵심 리소스 생성](docs/aws-deployment/step-42-core-resource-provisioning.md).
- 45단계 시크릿 주입, 46단계 테스트 통합, 47단계 품질 게이트 구성 완료.
- 21단계 아티팩트 전략 문서.

## 필요한 입력 자료
- 아티팩트 업로드 스크립트 또는 파이프라인 단계 정의.
- 무결성 검증 도구(sha256sum, cosign, Notary 등) 설정.
- 접근 정책 및 만료 정책, 보관/보존 요구사항.

## 상세 절차
1. **업로드 프로세스 정의**
   - 빌드 성공 이후 실행할 업로드 단계의 조건과 재시도 정책을 정의한다.
   - 아티팩트 네이밍 규칙과 메타데이터(버전, 커밋 SHA, 빌드 번호)를 확정한다.
2. **보안 및 인증 구성**
   - 45단계에서 구성한 시크릿을 사용하여 저장소 인증을 수행한다.
   - 업로드 시 TLS, 서명, 암호화가 적용되었는지 확인한다.
3. **무결성 검증 자동화**
   - 업로드 전/후에 해시 또는 서명 검증을 실행하여 결과를 로그로 남긴다.
   - 실패 시 파이프라인을 중단하고 자동으로 알림을 전송한다.
4. **메타데이터 및 태그 적용**
   - 아티팩트에 빌드 파라미터, 환경, 팀 정보를 태그로 추가한다.
   - 37단계 태깅 규칙과 비용 추적(36단계)을 고려한 메타데이터를 포함한다.
5. **접근 제어 검증**
   - 최소 권한 IAM 역할로 업로드와 다운로드 테스트를 수행한다.
   - 퍼블릭 접근이 차단되었는지, 필요 시 서명 URL을 통한 제한적 접근만 허용되는지 확인한다.
6. **감사 및 보존 설정**
   - 업로드 로그를 CloudTrail, CloudWatch Logs 또는 외부 로깅 시스템에 저장한다.
   - 보존/만료 정책을 설정하고 39단계 백업/DR 전략과 정합성을 확인한다.

## 의사결정 포인트
- 서명(코드 서명, 컨테이너 서명)을 필수로 할지 여부.
- 아티팩트 보존 기간과 자동 정리 정책.
- 재배포/롤백 시 이전 버전을 유지할 범위.

## 체크리스트
- [ ] 업로드 단계가 파이프라인에 통합되고 성공적으로 실행된다.
- [ ] 해시/서명 등 무결성 검사가 자동으로 수행된다.
- [ ] 접근 제어 및 태깅 정책이 적용되었다.
- [ ] 업로드 로그와 메타데이터가 감사 가능하도록 저장되었다.

## 산출물 및 보관 위치
- 아티팩트 업로드 스크립트 (`ci/scripts/upload-artifacts.sh`).
- 무결성 검증 보고서 (`docs/reports/artifact-integrity.md`).
- 저장소 접근 테스트 로그 (`docs/reports/artifact-access-validation.md`).
