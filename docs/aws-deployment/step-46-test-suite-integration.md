# 46단계. 테스트 스위트 통합

## 목적
- CI 파이프라인에 단위, 통합, 엔드투엔드 테스트를 통합하여 코드 품질을 자동으로 검증한다.
- 테스트 실행 결과가 26단계 모니터링과 31단계 품질 기준에 반영되도록 구성한다.

## 선행조건
- 27단계 파이프라인 요구사항 문서, 30단계 빌드 파이프라인 설계.
- [43단계 CI 인프라 구성](docs/aws-deployment/step-43-ci-infrastructure-setup.md) 및 [44단계 리포지토리 연동](docs/aws-deployment/step-44-repo-ci-integration.md).
- 45단계 빌드 시크릿 주입 설정.

## 필요한 입력 자료
- 테스트 프레임워크 설정 파일(Jest, pytest, JUnit 등)과 스크립트.
- 테스트 데이터/시드 파일, 모의 서비스(Mocking) 구성.
- 테스트 결과 보고서 포맷, 커버리지 목표, 알림 채널 정보.

## 상세 절차
1. **테스트 범위 및 우선순위 확정**
   - 0단계 요구사항과 67단계 기능 검증 계획을 참조하여 필수 테스트 목록을 정한다.
   - 단위/통합/E2E 단계별 실행 순서와 병렬 실행 가능 여부를 결정한다.
2. **파이프라인 스테이지 설계**
   - 빌드 단계 후 단위 테스트, 통합 테스트, E2E 테스트 순으로 스테이지를 배치한다.
   - 실패 시 파이프라인을 중단하고 34단계 승인 절차와 연계한다.
3. **테스트 데이터 및 시크릿 구성**
   - 45단계에서 설정한 시크릿을 사용하여 테스트 환경 자격 증명을 주입한다.
   - 통합/E2E 테스트를 위한 임시 인프라(예: 테스트 DB, Mock API)를 자동으로 생성/정리한다.
4. **커버리지 및 리포트 통합**
   - 커버리지 도구(istanbul, JaCoCo 등)를 사용해 결과를 산출하고 기준치를 설정한다.
   - 보고서를 S3/ECR 아티팩트 또는 테스트 관리 도구에 업로드한다.
5. **모니터링 및 알림 연동**
   - 테스트 실패 시 26단계 모니터링 도구나 Incident 채널로 알림을 전송한다.
   - 실행 시간, 실패율을 지표로 수집하여 90단계 KPI 추적에 활용한다.
6. **유지관리 프로세스 정의**
   - 테스트 케이스 추가/수정 절차, 코드 리뷰 기준을 문서화한다.
   - 정기적으로 오래된 테스트나 flaky 테스트를 점검하는 일정을 설정한다.

## 의사결정 포인트
- 통합/E2E 테스트를 어디까지 CI에서 실행할지 vs. 별도 환경에서 실행할지 여부.
- 테스트 환경 프로비저닝을 온디맨드로 할지 상시 유지할지 결정.
- 테스트 실패 허용 기준과 재시도 정책.

## 체크리스트
- [ ] 테스트 스위트가 파이프라인에 통합되어 자동 실행된다.
- [ ] 시크릿/테스트 데이터가 안전하게 주입되고 정리된다.
- [ ] 커버리지 및 테스트 보고서가 저장·공유된다.
- [ ] 실패 알림 및 모니터링 지표가 구성되었다.

## 산출물 및 보관 위치
- 테스트 실행 스크립트 (`package.json` 스크립트, `scripts/tests/*.sh`).
- 파이프라인 테스트 단계 정의 (`ci/pipelines/test-stage.yaml`).
- 테스트 결과 보고서 및 대시보드 링크 (`docs/reports/test-automation-summary.md`).
