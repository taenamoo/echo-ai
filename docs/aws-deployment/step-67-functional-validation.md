# 67단계. 애플리케이션 기능 검증

## 목적
- 배포된 애플리케이션이 비즈니스 요구사항을 충족하는지 기능 및 통합 테스트를 수행한다.
- 0단계 요구사항과 31단계 품질 기준을 기준으로 성공/실패를 판정한다.

## 선행조건
- [65단계 최초 배포](docs/aws-deployment/step-65-initial-dev-deployment.md) 및 [66단계 헬스체크 구성](docs/aws-deployment/step-66-healthcheck-configuration.md) 완료.
- [60단계 설정 값 검증](docs/aws-deployment/step-60-configuration-validation.md)과 [62단계 네트워크 테스트](docs/aws-deployment/step-62-network-connectivity-test.md) 결과가 정상이어야 한다.
- 테스트 케이스와 데이터가 [28단계 CI 도구 선정](docs/aws-deployment/step-28-ci-tool-selection.md), [46단계 테스트 스위트 통합](docs/aws-deployment/step-46-test-suite-integration.md)에서 정비되어야 한다.

## 필요한 입력 자료
- 기능/통합 테스트 케이스 목록, 테스트 자동화 스크립트.
- QA 시나리오, 사용자 스토리, 요구사항 추적 매트릭스.
- 테스트 데이터, 목(mock) 서비스 설정.

## 상세 절차
1. **테스트 범위 및 우선순위 확정**
   - 핵심 사용자 플로우, 외부 연동, 장애 시나리오 등을 포함하도록 테스트 범위를 정의한다.
   - 68단계 성능 테스트와 중복되지 않도록 목표를 구분한다.
2. **테스트 환경 준비**
   - 테스트 계정, 권한, 데이터 초기화 스크립트를 준비한다.
   - 외부 API가 필요한 경우 샌드박스 환경 또는 모킹 전략을 선택한다.
3. **자동화/수동 테스트 실행**
   - CI/CD에 통합된 자동 테스트를 실행하고 결과를 수집한다.
   - 수동 테스트가 필요한 경우 체크리스트에 따라 수행하고 증적(스크린샷, 로그)을 확보한다.
4. **결과 분석 및 결함 관리**
   - 실패 케이스를 버그 트래킹 시스템에 등록하고 심각도/우선순위를 지정한다.
   - 재테스트 일정을 계획하고 패치 전략을 수립한다.
5. **승인 및 보고**
   - 테스트 요약 보고서를 작성하여 이해관계자 승인을 받는다.
   - 34단계 승인 절차에 따라 다음 단계 진행 여부를 결정한다.

## 의사결정 포인트
- 기능 검증을 dev에서 완료할지, 별도 QA/스테이징 환경에서 재실행할지.
- 자동화 대비 수동 테스트 비율을 어떻게 유지할지.
- 결함 기준선을 어떻게 설정할지(예: P1 이슈 0건일 때만 승인).

## 체크리스트
- [ ] 모든 필수 기능 테스트 케이스가 실행되었다.
- [ ] 실패 케이스가 버그 트래커에 등록되고 우선순위가 지정되었다.
- [ ] 테스트 결과 보고서와 승인 기록이 확보되었다.
- [ ] 재테스트 또는 추가 개선이 필요한 항목이 명확히 정의되었다.

## 산출물 및 보관 위치

- 버그 트래킹 이슈 목록.
- 테스트 자동화 로그 및 스크린샷.
