# Step-00 요구사항 명세 검토 이행 기록

## 0. 선행조건 및 배경
- 프로젝트 킥오프가 완료되어 있으며, 현재 파악된 이해관계자는 개발자 3명이다.
- AWS 개발 서버 배포는 요청자가 단독으로 수행할 예정이다.
- 최신 제품 기획 및 기술 문서는 모두 동일 저장소(`docs/`)에서 확인 가능하며, 추가 외부 자료는 존재하지 않는다.
- 인프라는 CloudFormation 스택으로 관리할 예정이며, API Gateway–Lambda–DynamoDB(or S3) 조합으로 백엔드 API를 구성하고 정적 페이지는 S3 혹은 CloudFront로 배포할 계획이다.

### 사전 질문 및 응답
- **질문**: 최신 기획 문서, 조직도, 기존 RACI 매트릭스 등 추가 참고 자료가 있는가?
  - **답변**: 별도 자료는 없으며, 현재 이해관계자는 개발자 3명뿐이다.
- **질문**: AWS 개발 서버 배포 책임자는 누구인가?
  - **답변**: 배포 작업은 현재 담당자(요청자)가 단독으로 진행할 예정이다.

## 1. 입력 자료 수집 현황 (절차 1)
- 저장소 내 제품 요구사항 문서, 사용자 스토리 및 인수 기준, 시스템 아키텍처, 데이터베이스 스키마 문서를 확보하여 참고했다.
- 외부 저장소나 별도 산출물이 없으므로, 향후 요구사항 변경이나 추가 자료는 본 저장소에 신규로 축적해야 한다.

## 2. 요구사항 분류 결과 (절차 2)

### 기능 요구사항 (우선순위 High)
1. **인증/인가**: 이메일·비밀번호 기반 회원가입 및 로그인, 비밀번호 정책 준수, 자동 로그인 처리.
2. **문서 업로드**: `.txt`/`.md` 단일 파일 업로드, 업로드 진행 상태 피드백.
3. **문서 목록 및 상세 열람**: 요약 미리보기, 상세 화면에서 요약/원문 일부 제공, 삭제 시 사용자 확인.
4. **AI 요약 제공**: 업로드된 문서를 자동으로 요약하고 요약 상태를 추적.

### 비기능 요구사항 (우선순위 High~Medium)
1. **성능**: 업로드 후 30초 이내에 요약 결과 제공.
2. **보안**: 비밀번호 해시 저장, Secrets Manager 기반 API 키 관리, 최소 권한 원칙 준수.
3. **플랫폼 제약**: AWS 서버리스(Next.js + Lambda + DynamoDB) 아키텍처 유지.
4. **운영**: 요약 실패/지연 시 재시도 전략 및 모니터링 체계 필요.
5. **배포 자동화**: GitHub Actions에서 `develop`/`master` 브랜치 머지 시 CloudFormation 스택 배포를 트리거하며, UI 변경만 있을 때는 CloudFront로의 단독 배포 프로세스를 별도로 정의해야 한다.

## 3. 추가 정보 확인 및 미해결 사항 (절차 3)
- 요구사항 문서 외 참고 자료가 없음을 이해관계자에게 확인했다.
- 인프라 구성 방안(CloudFormation 기반 서버리스 스택, GitHub Actions 배포 트리거)을 확인했으며, UI 전용 배포 경로는 아직 미결 상태다.
- 파일 형식·요약 엔진·UI 세부 기준과 같이 모호한 항목은 향후 워크숍 또는 Q&A 세션에서 확정해야 한다.

## 5. 검토 회의 및 승인 현황 (절차 5)
- 추가 이해관계자가 없어 정식 회의는 미실시 상태이며, 추후 Q&A/워크숍을 통해 승인 절차를 마련해야 한다.
- RTM 초안은 저장소 내에서 버전 관리되고 있으며, 확정 시 `docs/requirements/rtm.xlsx` 형태로 확장 계획이다.

## 6. 현행 구현 대비 진단 요약
- **인증**: 비밀번호 해시와 JWT 발급은 구현되어 있으나 DynamoDB `SK` 패턴이 문서 정의(`PROFILE`)와 불일치하여 스키마 정합성 검증 필요.
- **문서 업로드**: 다중 파일 및 PDF/이미지 허용으로 요구사항(단일 `.txt/.md`)과 상충. 업로드 진행 표시 로직은 존재하나 제약 조건 재확인 필요.
- **문서 목록/상세**: 목록은 요약 상태를 표시하지만 삭제 확인 대화상자가 없고 상세 화면에 원문 일부가 노출되지 않아 인수 기준 미충족.
- **AI 요약**: 요구사항상 ChatGPT 사용 예정이나 구현은 Gemini SDK와 45초 타임아웃을 사용. SLA(30초) 검증 및 엔진 선택에 대한 승인이 필요.
- **보안/구성**: 비밀번호 해시 요건은 충족하나 Secrets Manager는 미적용(환경 변수 사용). API 키 관리와 환경 분리 전략이 추가 정의되어야 한다.

## 8. 오픈 이슈 및 확인 필요 사항
1. **파일 형식 제약**: `.txt/.md` 단일 업로드 요구사항을 유지할지, 현행 PDF/이미지 다중 업로드를 인정할지 결정 필요.
2. **요약 엔진 승인**: ChatGPT 사용을 강제할지, Gemini 사용을 요구사항 변경으로 승인할지 이해관계자 합의 필요.
3. **UI/UX 보완**: 삭제 확인 대화상자 및 상세 화면 원문 일부 노출 범위, 민감 정보 마스킹 기준 정의 필요.
4. **배포 전략 세분화**: CloudFormation을 통한 공통 스택 배포 외에, UI만 수정될 때 CloudFront/S3로 신속히 배포하는 절차를 확정해야 한다.
5. **데이터 모델 정합성**: DynamoDB `SK` 포맷, 요약 상태 필드(`summaryStatus` vs `status`)를 문서와 일치시키거나 문서 업데이트 필요.
6. **보안 운영**: Secrets Manager 적용 범위, 키 순환 절차, 환경 별 구성 전략 수립.
7. **리소스 제약 반영**: 이해관계자가 개발자 3명으로 제한되어 있으므로 RTM 담당자 지정 및 승인 절차를 간소화하는 방안 마련 필요.

## 9. 다음 단계 제안
1. **요구사항 확정 워크숍**: 개발자 3인과 배포 담당(요청자) 간 짧은 세션을 열어 기능/비기능 요구사항의 변경 여부를 확정하고 승인 프로세스를 정의한다.
2. **RTM 정식화**: 확정된 요구사항을 바탕으로 RTM(예: `docs/requirements/rtm.xlsx`)을 작성하고 담당자·검증 단계를 매핑한다.
3. **보안·인프라 계획 수립**: Secrets Manager 통합, Lambda 타임아웃 최적화, 배포 전략(단독 운영 고려)을 포함하는 초기 운영 계획서를 작성한다.
4. **배포 체계 명세화**: CloudFormation 중심 배포 파이프라인과 UI 전용 배포 플로우(CloudFront/S3)를 정의하고, GitHub Actions 워크플로에 반영한다.
5. **산출물 저장 전략**: 추가 자료가 없는 만큼, 향후 Q&A와 회의록을 `docs/requirements` 혹은 `docs/aws-deployment` 하위에 정리하여 지속 관리한다.
